VCube (Virtual Cube)

O presente trabalho tem como objetivo a implementação do algoritmo distribuído hierárquico VCube (Virtual Cube), um algoritmo para determinação do estado de todos os processos do sistema. Neste algoritmo os processos são organizados em clusters sob o ponto de vista do processo testador, de maneira que um cluster é testado a cada rodada de testes.

O algoritmo VCube foi implementado em duas versões. A principal vantagem da segunda versão do algoritmo é a diminuição no número de testes realizados para a identificação de um diagnóstico.



A figura acima representa a topologia virtual criada pelo algoritmo VCube quando o sistema possui oito processos sem falha. O hipercubo possui diversas propriedades logarítmicas extremamente importantes para sistemas que necessitam de escalabilidade.

Pontos-Chave

  • Segue o modelo de falhas crash;
  • Monitoramento baseado em testes;
  • O sistema deve ser síncrono;
  • Os canais de comunicação são perfeitos e jamais falham;
  • Sistema fully-connected.

Função C(i, s)

A função C(i, s) é essencial para a construção do algoritmo VCube pois a partir dela é possivel a organização dos processos em clusters. Sua saída retorna a lista de processos (em ordem) que devem ser testados pelo testador i nos clusters s = 1, 2,…, logN.

Número de Testes

Uma das principais características da segunda versão do algoritmo VCube é a diminuição do número de testes executados para a relização de um diagnóstico. Esta versão torna o VCube um algoritmo escalável, característica extremamente importante em sistemas distribuidos.

O pior caso do número de testes no VCube (versão 01) ocorre quando todos os processos do maior cluster (com N/2 processos) estão falhos. Os N/2 testadores testam todos os N/2 falhos, isto é, N/2 vezes N/2 que resulta em N²/4.

Na imagem abaixo podemos observar que os processos 4, 5, 6 e 7 estão falhos, com isso o número de testes seria de: 8²/4 = 16. Podemos visualizar que cada processo (0, 1, 2 e 3) correto executa 4 testes, totalizando o total de 16 testes.

Detalhes de Implementação

Para as Duas Versões

  • TIME_INTERVAL
    • Intervalo de tempo constante em que os processos executam testes periodicamente.
  • MAX_TIME
    • Tempo limite para que nossa simulação rode.
  • Rodada de Testes
    • Intervalo de tempo em que todo processo sem-falha testou pelo menos um de seus clusters. Cada processo testa um cluster por rodada.
  • Diagnóstico
    • Tempo necessário para que após a ocorrência de um evento, todos os processos corretos saibam do evento.
  • Latência do Diagnóstico
    • Número de rodadas de testes que devem ser executadas para completar o diagnóstico.

Vetor STATE

É utilizado para guardar as informações de diagnóstico dos demais processos. Seus possiveis estados são:

  • DESCONHECIDO (-1) -- Não se sabe o estado atual do processo.
  • SEM FALHA (0) -- O processo está ativo/correto.
  • COM FALHA (1) -- O processo encontra-se inativo.

Versão 01

Em cada intervalo de testes, cada processo correto executa testes em 1 cluster sequencialmente até encontrar um processo correto. Se o processo testador encontrar algum processo correto, o testador obtem informações sobre todos os outros processos do clusters (os quais ele não testou). Se o processo testador não encontrar nenhum processo correto dentro do cluster, a rodada de testes é finalizada.

Versão 02

Nesta versão, a cada intervalo de testes, o vetor TESTE é atualizado. Quando o processo atual (token) entra em uma rodada de testes, ele verificará para cada processo se ele é o testador do processo em questão ou não. Caso ele seja, ele testará o processo. Caso ele não seja, ele passará para o próximo processo.

Vetor TESTE

É utilizado para guardar a informação sobre quais processos o nodo deverá testar. Seu tamanho é definido pelo N (número de processos do sistema) e seus possíveis valores são:

  • 0 -- Não teste o processo
  • 1 -- Teste o processo

Cada índice do vetor representa um processo do sistema. Por exemplo, para um sistema com 4 processos considere o nodo 0 que possui o seguinte vetor TESTE: 0 0 0 1. Este vetor representa que o processo 0 deverá testar o processo 3 e não deverá testar o processo 0 (ele mesmo) e os processos 1 e 2.
